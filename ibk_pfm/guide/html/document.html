퍼블리싱 가이드

작성규칙

1. HTML

  • i-ONE뱅크 모바일은 웹표준/웹접근성을 준수
  • iOS 8.0 이상 / Android 4.4 이상 (서비스별로 상이할 수 있음)
  • 디자인적으로 치명적이지 않은 1~2px의 차이는 무시

1-1. 구조와 표현

  • 구조언어 - HTML, XHTML, XML : 웹의 사용자 환경에서 구조(Structure)를 나타내는 언어
  • 표현언어 - CSS : 웹의 사용자 환경에서 표현(Rendering)을 나타내는 언어
  • 마크업 언어(HTML)와 화면 표시(CSS)언어를 본래의 목적에 맞게 분리
  • 모든 요소(Element)와 속성(Properties) 및 값(Value)은 영문 소문자
  • 속성값은 큰 따옴표(“”)로 묶음
  • W3C Validation
    • HTML 문법 검사 : http://validator.w3.org/
    • CSS 문법 검사 : http://jigsaw.w3.org/css-validator/

1-2. 특수문자 사용 HTML Encoding

특수 기호는 문자 엔티티 참조하여 코드로 변환한다.

  • - quot [ " ] &#34;
  • - number [ # ] &#35;
  • - percent [ % ] &#37;
  • - amp [ & ] &#38;
  • - apos [ ' ] &#39;
  • - asterisk [ * ] &#42;
  • - lt [ < ] &#60; / &lt;
  • - gt [ > ] &#62; / &gt;
  • - middot [ · ] &#183;
  • - ₩ [ ₩ ] &#8361;

1-3. 기본구조

  • 전체 문서는 <html> ~ </html> 안에 들어간다.
  • <head> ~ </head> 안에는 문서의 기본정보가 들어가는 공간이고 화면에 표시되지 않는다.
  • head 요소에는 title, meta 요소 등이 포함된다.
  • <body> ~ </body> 안에는 실제 화면에 표시되는 내용이 들어간다.
  • ※ 코드의 가독성 및 HTML 구조 파악의 용이성을 위해 들여쓰기를 한다. 들여쓰기할 때는 tab(4칸)을 사용한다.
  • ※ HTML 코드의 빈 줄은 코드 그룹의 영역을 표시하기 위하여 사용한다.
  • HTML5 - HTML문서는 반드시 DTD(Document Type Definition)를 선언한다.
  • 언어코드(lang)는 User Agent가 언어를 올바로 해석할 수 있게 도와주며 검색과 음성 장치에 활용된다.
  • utf-8은 다국어 지원이 가능하며 euc-kr보다 표현 가능한 한글이 많다.
  • HEAD 작성 순서 - meta, title, link, script, style 순서로 선언
  • 속성값 작성 순서 - rel, type, href/src, width/height, target, id, name, class, style, title/alt 그 외 속성들 순서로 선언
  • viewport - <meta name="viewport" content="width=device-width, user-scalable=no, initial-scale=1.0, minimum-scale=1.0, maximum-scale=1.0, viewport-fit=cover">

1-4. 주석표기

  • 시작주석: <!-- 주석내용 -->, 종료주석: <!-- //주석내용 -->
  • 코드의 가독성을 높이고 전체 HTML 구조를 쉽게 파악하기 위하여 주석 사용 준수
  • 레이아웃 구분 : HTML 문서의 구조 파악에 도움이 되게끔 시작과 끝 부분을 주석으로 표시
  • 수정작업 시 주석처리 : 마크업 수정 시 수정부분을 표기하는 주석은 날짜를 기입하여 사용
<!-- YYYY-MM-DD 주석내용 --> <div> ... </div> <!-- //YYYY-MM-DD 주석내용 -->

3. 앱접근성

지침 1. 대체 텍스트

  • 이미지 등 텍스트 아닌 콘텐츠를 제공할 경우에는 반드시 대체 텍스트를 함께 제공해야 한다.
  • 대체 텍스트는 가능한 짧고 명확(short & clear)하게 제공하는 것이 바람직하며,
    보이스오버, 토크백 등 모바일 스크린리더에서 기본적으로 제공해 주는 용어인 “버튼”, “이미지”, “레이블” 등은 중복하여 제공하지 않는 것이 바람직하다.
  • ※ 웹 접근성 연구소 버튼(×) → 웹 접근성 연구소(○)
  • ※ 모바일 접근성 이미지(×) → 모바일 접근성(○)
  • ※ 금융자동화기기 접근성 레이블(×) → 금융자동화기기 접근성(○)

지침 7. 초점 - 검사항목 7.1. 초점은 사용자가 예측할 수 있도록 논리적인 순서로 이동해야 한다.

  • 하위 메뉴는 상위 메뉴 바로 다음에 와야 하며, 계층관계가 명확하도록 마크업 되어야 함.
  • 폼 콘트롤 요소와 연계된 <label>은 폼 콘트롤 요소에 인접(바로 옆에)해야 함.
  • 게시판 형태의 UI는 테이블로 마크업하고 <th> scope를 제공하는 것이 가장 바람직함.
  • 테이블 형태로 마크업하기 어려운 경우는 콘텐츠를 선형화했을때 '11.12.02', '4385' 등의 값이 각각 '작성일', '조회수'임을 바로 알 수 있도록 숨김텍스트 등을 추가로 제공해야 함.
  • 탭 형태의 콘텐츠에서 각 탭에 해당하는 콘텐츠가 무엇인지 알 수 없는 경우 논리적인 순서에 맞게 숨김텍스트 등을 추가로 제목 제공.
  • 아이디/비밀번호 입력창에서 콘텐츠가 논리적인 순서 (아이디 → 비밀번호 → 로그인 상태 유지 → 로그인 버튼)로 배치.

지침 8. 누르기 동작 지원

  • 터치(touch) 기반 모바일 기기의 모든 컨트롤은 누르기 동작으로 제어할 수 있어야 한다.
  • 다중 누르기(Multi-touch), 슬라이드(Slide), 끌기와 놓기(Drag and drop) 등의 복잡한 동작을 사용하는 기능이 있는 경우 해당 기능을 대체할 수 있는 단순한 누르기 동작을 지원.

지침 12. 입력 도움 - 검사항목 12.1. 입력서식에는 용도와 목적을 알 수 있는 대체정보를 제공해야 한다.

  • 스크린리더나 점자정보단말기, 모바일의 VoiceOver, TalkBack 등은 폼콘트롤요소의 id값과 label의 for값이 연결되어 있는 경우 폼요소 선택 시에 연계된 label을 읽어줌으로써 폼 콘트롤 요소를 인식할 수 있음.
  • <label>은 눈에 보이지 않게 제공할 수 있고, 폼 콘트롤 요소에 title로도 제공 가능함.
  • * label이 제공되기 어려운 경우에 한하여 title을 인정해 주고 있으며 placeholder는 고려 대상이 아님.(감점)
label[for] = input[id]
  • <label>의 for 값과 폼요소의 id 값을 동일하게 제공하여 연결.
  • 1:1에 대응하는 레이블을 제공할 수 없는 경우에는 title 속성을 이용하여 폼 요소의 설명을 제공해야 함.
  • <label for="nickname"> 닉네임</label> <input id="nickname" name="nickname" type="text" value="홍길동">
  • label 제공하기 힘들 경우 title 속성으로 대체
  • <th scope="row"><label for="tel01">전화번호</label></th> <td> <input type="text" id="tel01" title="전화번호 첫째자리"> - <input type="text" id="tel02" title="전화번호 둘째자리"> - <input type="text" id="tel03" title="전화번호 셋째자리"> </td>
